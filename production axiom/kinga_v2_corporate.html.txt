  <div class="view" id="view-spec">
    <div class="topbar">
      <div><div class="crumbs">REFERENCE</div><h1>Functional Specification</h1>
        <div class="topbar-sub">What every screen in this prototype represents in the real product — excluding backend and database implementation, as requested.</div>
      </div>
    </div>

    <div class="spec-box"><div class="spec-label">01 — Portfolio</div><div class="spec-text">A live, ranked list of every client a firm manages, ordered by open exposure. Each row reflects the most recent reconciliation run for that client. A "Clean" status means no variance exceeded the configured materiality threshold on the last run — not that no data exists.</div></div>

    <div class="spec-box"><div class="spec-label">02 — Findings</div><div class="spec-text">Each finding is generated by one of two mechanisms: (a) a reconciliation diff between two data sources (ledger vs EFDMS/EFD), or (b) a rule trigger fired by a ledger entry matching a statutory category (e.g. a "rent" line triggering a WHT-on-rent check). Every finding stores its source records, so a user can trace the number back to the original transactions, not just see a final figure.</div></div>

    <div class="spec-box"><div class="spec-label">03 — Evidence Workflow</div><div class="spec-text">Each finding has its own six-step tracked process: request, awaiting client, received, preparer review, partner sign-off, submitted. Document requests are sent to the client (email/portal), reminders fire automatically on a schedule, and every status change is timestamped. This is the layer that creates switching cost — the accumulated history of requests and responses lives here, not in the reconciliation logic itself.</div></div>

    <div class="spec-box"><div class="spec-label">04 — Rules Engine</div><div class="spec-text">A versioned, effective-dated table mapping ledger categories and document types to statutory obligations. Rules can be extended per industry (a "tourism levy" rule pack for safari operators, a "medical stock VAT" pack for pharmacies) without changing the core engine — new rows, not new code.</div></div>

    <div class="spec-box"><div class="spec-label">05 — Response Pack</div><div class="spec-text">A structured draft, generated only once all evidence linked to a finding is marked "Received" in the Workflow tab. This sequencing is a deliberate control: the system should not allow a response to be exported while evidence is still outstanding, since an incomplete response is itself a risk.</div></div>

    <div class="spec-box"><div class="spec-label">Roles &amp; Access (not shown as a separate screen)</div><div class="spec-text">Three roles run through every screen: Preparer (drafts findings and responses), Partner (sign-off required before submission to TRA), and Client (can only see and upload against their own outstanding document requests — they do not see the firm's other clients or internal notes).</div></div>

    <div class="spec-box"><div class="spec-label">What this prototype deliberately does not show</div><div class="spec-text">No predictive risk scoring, no "audit readiness" percentage, and no audit simulation. Per the design review this build is based on, those features create unsupportable liability if wrong and are deferred until the deterministic reconciliation and workflow layers are proven with real paying clients.</div></div>
  </div>
